# 漢字テストメーカー (Kanji Test Maker), project plan

A web tool for teachers to turn pasted Japanese sentences into printable
vertical-writing kanji worksheets (like `13. 絵・知・春・思　練習.pdf`),
output as an editable Word .docx.

---

## 1. Requirements (from your spec)

1. A page with a large text editor where a teacher pastes sentences.
2. Sentences are processed in JS and shown one per row in a table, with the
   kanji words auto-selected. The teacher can toggle the selection of any word.
3. A button generates a **.docx** with A4 landscape sheets. Options such as
   number of sentences per page.
4. On each page: sentences written **vertically**; one **rectangle per marked
   word** (sized to the number of characters to write) placed next to the
   sentence; each marked word **replaced by its reading**, underlined
   (side-lined in vertical writing). See the sample PDF.
5. A **kaki / yomi switch per sentence**:
   - *kaki* (write): marked word shown as its reading + an empty box to write
     the kanji.
   - *yomi* (read): marked word kept as the kanji (tests whether the student
     can read it).
6. Hosting: ideally a simple deploy (GitHub or similar).

---

## 2. Feasibility (researched, with working proofs of concept)

Everything needed has been validated on this machine. Artifacts live in `poc/`.

| Risk | Verdict | Evidence |
|---|---|---|
| Vertical (縦書き) layout, RTL columns | **Confirmed** | `poc/min.pdf`, a docx generated by JS and rendered by LibreOffice shows A4 landscape, three right-to-left vertical columns, correct vertical punctuation. |
| Same layout as crisp PDF (for preview / optional export) | **Confirmed** | `poc/poc.pdf`, headless Chrome from CSS `writing-mode: vertical-rl`, already close to the sample including inline boxes. |
| Generating .docx from the browser | **Confirmed** | The `docx` JS library (v9) emits valid OOXML with `textDirection=tbRl`; runs fully client-side. |
| Kanji → reading (to hiragana-ise / detect kanji words) | **Confirmed approach** | kuromoji.js, a pure-JS morphological analyzer with a bundled dictionary, runs in the browser, no server. |
| Hosting | **Static is enough** | All processing and .docx generation are client-side JS, so it can be a static site (GitHub Pages). PHP/server not required. |

Two gotchas found and already solved in the spike:
- Set page **orientation only** (do not also pass an explicit width/height, that
  produced a portrait page with a landscape flag).
- Vertical table cells **need explicit row height and cell width**, or the
  vertical content collapses to nothing.

Still to prove in implementation (not blockers, standard docx):
- The **answer boxes** as bordered elements inside a vertical cell (nested
  bordered table or a bordered frame). Renders fine in principle; needs the
  same explicit-sizing care, and a check in real Word, not only LibreOffice.

---

## 3. Recommended architecture

A **static, client-side web app**, no backend:

```
[ big paste box ]
      | sentences (one per line)
      v
[ kuromoji tokenizer ]  -> per sentence: tokens, each with reading + has-kanji
      v
[ editable table ]  one row per sentence:
      - tokens rendered; kanji-bearing words auto-selected (highlighted)
      - click a word to toggle its selection
      - per-row kaki / yomi switch
      - (live mini preview of that sentence's vertical form)
      v
[ options ]  sentences per page, font, header fields (class, lesson no., title)
      v
[ DOCX builder (docx.js) ]  vertical RTL columns + boxes, per kaki/yomi
      v
[ Download .docx ]   (+ optional "Download PDF" via the same layout in Chrome)
```

Why static / client-side:
- No copyright/storage concerns (sentences are the teacher's own).
- Trivial, free hosting (GitHub Pages or a folder on your server).
- kuromoji and docx both run in the browser; nothing to install for users.

PHP note: your spec said "PHP script ... or somewhere else". Since all the
work is JS, **PHP is not needed**; a static page is simpler and hostable on
GitHub Pages directly. (If you ever want server-side .docx, PHPWord exists, but
it would only add complexity here.)

## 4. Worksheet model (the tricky bits)

- **Word vs box.** A marked word may be several characters (絵画, 思い出す).
  The **box height = number of kanji to write**; **okurigana stays as kana**
  outside the box (思い出す → box(思)・い・box(出)・す, or a 2-cell box for 思出
  with い/す as kana, to be matched to the sample).
- **kaki mode (default):** in the sentence, the marked word is shown as its
  **reading in hiragana with a side line**; a box of the right size sits next
  to the sentence for writing the kanji.
- **yomi mode:** the marked word is shown **as the kanji**; the box next to the
  sentence is where the student writes the **reading** (the box is sized for the
  reading's kana count).
- **Auto-selection:** a "kanji word" = a kuromoji token containing at least one
  kanji. All such tokens start selected; the teacher deselects the ones that
  should stay plain. (Matches the sample, where only the lesson's kanji are
  tested and a few known kanji are left as kanji.)
- **Readings can be wrong** (context, 熟字訓): the table preview is **editable**,
  the teacher fixes a reading before generating. This keeps kuromoji errors off
  the printed sheet.

## 5. Tech stack

- Plain **HTML/CSS/JS**, no build step (easy to host and maintain). A light
  layer (Alpine.js or Preact) only if the table UI gets busy.
- **kuromoji.js** (or kuroshiro on top) for tokenizing + readings; the
  dictionary (~a few MB) is fetched once and cached.
- **docx** (dolanmiu) for .docx generation in the browser.
- **Font selector** (confirmed feature). The teacher picks from several bundled
  fonts, and can also **upload their own font file**. Behaviour differs by output:
  - **PDF export:** any chosen/uploaded font is loaded in the browser via the
    FontFace API and **embedded by Chrome**, so the PDF looks identical
    everywhere. This is the faithful path for custom fonts.
  - **.docx:** the `docx` library can only set a font **name**, it does not embed
    font binaries. So a font shows correctly only if it is **installed on the
    machine that opens the .docx**. Bundled choices should therefore default to
    fonts already present on typical Mac/Windows Japanese systems (Hiragino,
    Yu Mincho/Gothic); an uploaded custom font in .docx is name-only (the PDF
    export is the way to guarantee it).
  - Worksheets traditionally use 教科書体 (textbook style) for canonical stroke
    shapes; offer it as a bundled option if a licensed file is available
    (it will be faithful in PDF regardless of the opener's machine).

## 6. Decisions

Confirmed:
- **yomi mode** has a box for writing the **reading** (sized to the kana count).
- **PDF export** is included alongside .docx (a "Download PDF" button using the
  same layout; also the faithful path for custom fonts).
- **Font selector** with bundled fonts **and** custom font upload (see the font
  note in section 5 for the PDF-vs-docx behaviour).

Still to confirm (with my default):
1. **Hosting:** static site on GitHub Pages (recommended) vs your own server.
   *Default: GitHub Pages.*
2. **Box placement:** boxes grouped beside each sentence column in word order,
   matching the sample. Confirm against the sample once boxes are implemented.
3. **Okurigana / box granularity:** one box per kanji vs one merged box per word
   (the sample looks merged for compounds). *Default: match the sample
   (merged box sized to the kanji count, okurigana as kana outside).*
4. **Bundled font list:** which fonts to ship in the selector (defaulting to
   ones already installed on typical Mac/Windows JP systems for .docx), and
   whether to license a 教科書体. *Default: Hiragino/Yu Mincho + Gothic, plus a
   教科書体 if a licensed file is provided.*
5. **Header fields:** class code, lesson number (⑬), title, name field, all
   editable per sheet. *Default: yes, all editable.*

## 7. Phased plan

**Phase 0, spikes (done).** Vertical RTL layout proven in both .docx
(`poc/min.pdf`) and PDF (`poc/poc.pdf`); docx.js + kuromoji approach confirmed.

**Phase 1, core generator (MVP). — DONE (working, browser-tested).**
- Paste box, split into sentences.
- kuromoji tokenize; render the editable table with auto-selected kanji words,
  toggle, and per-row kaki/yomi switch; editable readings.
- Options: sentences per page; header fields.
- Shared layout model used by both exporters.
- docx.js builder: vertical RTL columns, reading-replacement with side line,
  answer boxes sized per word, kaki (box for kanji) vs yomi (box for reading).
- **PDF export** from the same layout (Chrome/CSS), reusing the proven path.
- **Font selector**: bundled fonts + custom upload (FontFace for preview/PDF;
  font name for .docx).
- Download .docx and PDF. Validate .docx in real Word (not just LibreOffice).

**Phase 2, fidelity & polish.**
- Match the sample exactly (box look, side lines, spacing, circled numbers,
  title row, name field).
- 教科書体 bundled option if a licensed file is available.
- Pagination across multiple pages; long-sentence wrapping.

**Phase 3, nice-to-haves (deferred).**
- Answer key, randomized variants (not requested now).
- Grade-based known-kanji presets to drive auto keep-vs-test.
- Save/load sheet definitions.

## 8. Risks & mitigations

- **DOCX vertical rendering varies by Word version.** Mitigate: explicit sizing
  (learned in the spike), test in real Word, and offer the proven PDF export as
  a fallback for faithful printing.
- **Reading accuracy (kuromoji).** Mitigate: editable readings in the table.
- **Font shapes on the opener's machine (.docx).** The .docx cannot embed a
  custom font, so it depends on what is installed where it is opened. Mitigate:
  default the bundled list to fonts already present on typical JP systems, and
  use the **PDF export** when exact appearance (or an uploaded font) matters.
- **Okurigana / compound box rules.** Mitigate: settle the rule against the
  sample early in Phase 1.
